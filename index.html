Snowflake has shipped a lot under the "Cortex" banner. If you've been confused about when to use Cortex Search versus Cortex Analyst versus Cortex Code — you're not alone. The names are similar. They all involve natural language. They all live inside Snowflake. But they are solving fundamentally different problems.

This guide breaks it down clearly: what each does, where each shines, real examples with code, and a decision framework you can actually use.

The Core Idea: Three Layers of Intelligence

Think of the Cortex suite as three distinct layers inside your Snowflake environment:

🔍
Layer 1
Cortex Search
Retrieves answers from unstructured text using semantic embeddings and keyword ranking. Think RAG without the plumbing.
Unstructured → Retrieval
📊
Layer 2
Cortex Analyst
Translates natural language into SQL on structured tables and semantic models. Powered by an LLM that understands your schema.
Structured → Analytics
⚙️
Layer 3
Cortex Code
An AI coding agent that builds your data infrastructure — dbt models, Airflow DAGs, Streamlit apps, Snowflake admin — from plain English.
Instructions → Code
The Golden Rule

The key decision is not the tool — it's the data type + what you're trying to do with it. Is your data in tables or in documents? Do you want to query it, retrieve from it, or build something on it?

Cortex Search — AI Over Unstructured Data

Cortex Search is Snowflake's managed RAG (Retrieval Augmented Generation) service. You point it at text columns — customer reviews, support tickets, call transcripts, legal documents, knowledge base articles — and it creates a search index using vector embeddings + keyword matching. Your users can then query it in plain English and get the most relevant passages back.

How it works under the hood

Figure 1 — Cortex Search indexes your text data automatically, then retrieves the most relevant chunks for any natural language query

Setting it up (it's just SQL)

-- Step 1: Create the search service on your text column
CREATE OR REPLACE CORTEX SEARCH SERVICE support_search
  ON ticket_text
  ATTRIBUTES category, priority
  WAREHOUSE = my_wh
  TARGET_LAG = '1 hour'
  AS SELECT
      ticket_id,
      ticket_text,
      category,
      priority
  FROM support_tickets;

-- Step 2: Query it in Python (REST / Snowpark)
response = root.databases['mydb'] \
    .schemas['public'] \
    .cortex_search_services['support_search'] \
    .search(
        query="payment failed but charged",
        columns=["ticket_text", "category"],
        limit=5
    )

Cortex Analyst — Natural Language to SQL

Cortex Analyst is for your structured data. It's an LLM-powered layer that sits on top of your Snowflake tables and semantic models, letting business users ask questions like "Which customers had the highest revenue last quarter?" — and getting back accurate, auditable SQL results.

The key ingredient is the Semantic Model — a YAML file you define once that maps your business terminology to your actual schema. "Revenue" maps to SUM(order_value). "Active users" maps to a specific filter. This is what makes Cortex Analyst accurate, not just fluent.

The Semantic Model (the secret sauce)

# semantic_model.yaml
name: sales_model
tables:
  - name: orders
    description: "All customer orders"
    base_table:
      database: prod
      schema: public
      table: orders

    measures:
      - name: total_revenue
        description: "Sum of order values"
        expr: SUM(order_value)
        data_type: number

    dimensions:
      - name: customer_segment
        description: "Customer tier: enterprise, mid-market, smb"
        expr: customer_segment

verified_queries:
  - name: top_customers_by_revenue
    question: "Which customers had the highest revenue last quarter?"
    sql: |
      SELECT customer_name, SUM(order_value) as revenue
      FROM orders
      WHERE DATE_TRUNC('quarter', order_date) = DATE_TRUNC('quarter', DATEADD('quarter',-1,CURRENT_DATE()))
      GROUP BY 1 ORDER BY 2 DESC LIMIT 10

Figure 2 — Cortex Analyst translates a business question through a semantic model into executable SQL, then returns results alongside the query for auditability

Why auditability matters

In regulated industries, you can't hand an executive an AI-generated number without showing your work. Cortex Analyst surfaces the SQL behind every answer — the business user sees the result and the query that produced it. That's trust.

Cortex Code — The AI Builder

Cortex Code is different from Search and Analyst. While those tools query your data, Cortex Code builds your data infrastructure. It's an AI coding agent that operates in Snowsight and as a CLI, and can generate production-ready SQL, Python, dbt models, Airflow DAGs, and Streamlit apps from plain English descriptions.

Think of it as having a senior data engineer pair-programming with you — one who knows your Snowflake schema, understands your semantic models, and can write diffs for your review before committing anything.

-- Cortex Code prompt → generated code (example output)
-- Prompt: "Create a table of customers at churn risk,
--          defined as no orders in 60 days + >2 support tickets"

CREATE OR REPLACE TABLE churn_risk_customers AS
WITH last_orders AS (
    SELECT
        customer_id,
        MAX(order_date) AS last_order_date,
        DATEDIFF('day', MAX(order_date), CURRENT_DATE()) AS days_since_order
    FROM orders
    GROUP BY 1
),
ticket_counts AS (
    SELECT customer_id, COUNT(*) AS ticket_count
    FROM support_tickets
    WHERE created_at >= DATEADD('day', -90, CURRENT_DATE())
    GROUP BY 1
)
SELECT
    lo.customer_id,
    lo.last_order_date,
    lo.days_since_order,
    tc.ticket_count,
    'HIGH' AS churn_risk_level
FROM last_orders lo
JOIN ticket_counts tc USING (customer_id)
WHERE lo.days_since_order > 60
  AND tc.ticket_count > 2;

Real-World Scenarios

1

Support team wants to find similar tickets — Use Cortex Search

Your support team gets a new ticket: "customer was charged twice after the app crashed." They want to find all previous tickets that sound similar.

"Find all tickets similar to 'double charge after crash' from the last 90 days"

Cortex Search uses vector embeddings to find semantically similar tickets — even ones that don't use the words "double charge" or "crash." It ranks by similarity and returns the top results with their original text.

💡
Why not Cortex Analyst? Because you're not asking for a count or an aggregate — you're asking for relevant text. There's no SQL that finds "semantically similar" support tickets without a vector index.
2

VP of Sales asks "why did revenue drop last month?" — Use Cortex Analyst

This is a structured data problem. Revenue lives in an orders table. A drop has a measurable cause — fewer orders, lower average order value, a specific customer segment declining.

"Show me revenue by segment this month vs last month, and flag any that dropped more than 10%"

Cortex Analyst translates this into SQL, runs it against your orders table, and shows the VP the result alongside the exact query — so they can verify it's right. Crucially, it also shows the SQL, which the analyst can tweak.

💡
Why not Cortex Search? Revenue is a number in a table, not a concept in a document. There's nothing to retrieve — you need aggregation, GROUP BY, and comparison logic. That's SQL territory.
3

Data engineer wants to build a churn model pipeline — Use Cortex Code

You want a repeatable dbt pipeline that calculates a churn risk score for every customer daily, integrating orders, support ticket history, and product usage.

"Build a dbt model called customer_churn_risk that joins orders, support_tickets, and product_events. Score 0-100 based on recency, ticket volume, and session frequency."

Cortex Code doesn't just write one SQL query — it scaffolds the entire dbt model with the right folder structure, ref() calls, tests, and documentation. It shows you a diff before applying anything.

💡
Why not Cortex Analyst? Analyst is for querying existing data. You want to build new infrastructure — a dbt model that runs daily and writes to a table. That's Cortex Code.
4

Customer success manager wants a full picture — Use all three via Cortex Agent

The manager wants to know: "Give me a full account briefing for Acme Corp before my renewal call — revenue trend, recent support issues, and any contract terms I should know."

"Account briefing for Acme Corp: revenue trend last 6 months, their top 3 support themes, and any renewal clauses from their contract"

This is a multi-step workflow. A Cortex Agent orchestrates it: Cortex Analyst handles the revenue query, Cortex Search retrieves support ticket themes and contract clauses, and the agent synthesizes everything into a briefing. None of these three can do it alone.

💡
Cortex Agent = orchestrator. It doesn't replace Search or Analyst — it routes to the right one based on what each step needs, then combines the results.

Common Mistakes to Avoid

  • ✗
    Using Cortex Search for analytics. "How many tickets were filed last week?" is a count query — it belongs in Cortex Analyst (or plain SQL). Search retrieves similar content, it doesn't aggregate.
  • ✗
    Using Cortex Analyst for document retrieval. "Find contracts that mention auto-renewal" is a semantic retrieval task. Analyst writes SQL — it can't do fuzzy text matching across free-form document content without a search index.
  • ✗
    Skipping the Semantic Model for Analyst. Without a semantic model, Cortex Analyst guesses at your schema and business definitions. "Revenue" might mean gross or net. Spend the hour to define it once — you'll get dramatically more accurate results.
  • ✗
    Using Cortex Agent for simple single-step queries. If you just need to ask a revenue question, call Analyst directly. The Agent orchestration layer adds latency and complexity — only reach for it when you genuinely need multi-step reasoning.
  • ✗
    Treating Cortex Code as a one-shot code generator. It's most powerful when used iteratively — describe the outcome you want, review the plan, then review the diffs. "Build a churn dashboard" is better than "write me a GROUP BY query."

Quick Decision Framework

Pick your tool in 3 questions
  • Is your data in text fields / documents? → Cortex Search
  • Is your data in structured tables and you want analytics? → Cortex Analyst
  • Do you want to build, automate, or scaffold infrastructure? → Cortex Code
  • Do you need structured + unstructured + reasoning in one flow? → Cortex Agent (uses Search + Analyst)
"These are not interchangeable features — they represent different layers of an AI-native architecture inside Snowflake."

The future of data platforms isn't just faster queries — it's building systems where intelligence operates natively across structured and unstructured data, with developers spending less time on boilerplate and more time on logic that matters.

Snowflake Cortex gives you all three layers. Now you know which one to reach for.
